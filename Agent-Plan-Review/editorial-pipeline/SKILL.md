# 三審三校 — 文章品質管控流程

> ⚠️ **核心原則：每個審校角色 = 獨立 subagent**
> 每位審校人員都必須以獨立 subagent 召喚，而非讓同一 agent 扮演多個角色。
> 這確保各輪審查視角真正彼此獨立，不受前一輪思路污染。

---

## 適用場景

- 部落格文章、技術教學、電子報發出前
- 長篇論述、RFC、產品介紹、品牌文案送審前
- 需要三方獨立視角確保論點嚴謹、文字精準的高能見度內容

---

## 流程總覽

```
產出草稿
   │
   ▼
【三審階段】— 內容層次審查
   ├── 初審 subagent：結構與完整性
   ├── 複審 subagent：邏輯、準確性、深度
   └── 終審 subagent：最終裁決、釋出可行性
   │
   ▼ （每輪回饋 → 修改 → 下一輪）
   │
【三校階段】— 細節層次校對
   ├── 初校 subagent：逐行比對，找排版/語法/命名問題
   ├── 二校 subagent：確認初校修正正確、無新引入錯誤
   └── 三校 subagent：作者視角最終確認，簽核定稿
   │
   ▼
定稿發布
```

## 三審詳細規格

### 初審 subagent — `content-first-reviewer`

**職責：** 把關基本品質門檻，不讓明顯破洞進入後續流程。

**checklist：**
- [ ] 論點清楚，有明確主旨句
- [ ] 段落結構：開頭、正文、結尾完整
- [ ] 是否符合目標讀者的知識水準與期待
- [ ] 有無空洞的填充句、AI 套話、過度修飾語

**輸出格式：**
```
初審結果（content-first-reviewer）
狀態：[通過 / 退修]
問題清單：
  - [嚴重] ...
  - [建議] ...
修改指示：...
```

---

### 複審 subagent — `content-deep-reviewer`

**前置條件：** 初審通過後才啟動複審。

**職責：** 深入挖掘邏輯一致性、內容準確性、潛在風險。

**checklist：**
- [ ] 論點是否有事實依據，避免自造數據或誇大宣稱
- [ ] 前後章節邏輯是否一致，無自相矛盾
- [ ] 每個段落是否都在推進核心論點，無離題段落
- [ ] 引用、數據、來源是否可信可查
- [ ] 語氣與立場是否全文一致（不混用正式/非正式）
- [ ] 是否有品牌/法律/版權潛在風險

**輸出格式：**
```
複審結果（content-deep-reviewer）
狀態：[通過 / 退修 / 需架構討論]
重大問題：...
一般問題：...
亮點（值得保留）：...
```

---

### 終審 subagent — `content-chief-reviewer`

**前置條件：** 複審通過後才啟動終審。

**職責：** 最終裁決，確認整體可發布。不重複前兩輪細節，聚焦全局判斷。

**checklist：**
- [ ] 整體聲音（voice）是否一致，讀起來像同一個人
- [ ] 標題與內文的承諾是否兌現（不標題黨）
- [ ] 是否達到預期讀者行動（理解、說服、點擊、訂閱）
- [ ] 結尾是否有力，不是自然消散
- [ ] 可否公開發布（無法律、品牌、版權疑慮）

**輸出格式：**
```
終審裁決（content-chief-reviewer）
裁決：[核准發布 / 退修 / 暫緩]
理由：...
條件（若有）：...
```

## 三校詳細規格

> 三校針對「已通過三審」的內容，聚焦細節錯誤，而非結構性問題。

### 初校 subagent — `proofreader-first`

**職責：** 對照最終修訂稿，逐段掃描細節錯誤。

**checklist：**
- [ ] 錯別字、同音異字、漏字
- [ ] 標點符號使用正確（句號、逗號、引號、破折號）
- [ ] 標題層次（H1/H2/H3）是否正確且一致
- [ ] 超連結文字與目標是否相符
- [ ] 數字格式一致（全形/半形、千分位、單位）
- [ ] 專有名詞、技術術語是否是常用縮寫且正確的寫法

**輸出格式：**
```
初校報告（proofreader-first）
錯誤清單（含行號或段落位置）：
  行 12：「...」→ 應改為「...」
  ...
```

---

### 二校 subagent — `proofreader-second`

**前置條件：** 初校修正已套用。

**職責：** 獨立確認初校修正是否正確執行，並找出初校遺漏或新引入的錯誤。

**checklist：**
- [ ] 初校中標出的問題是否都已修正
- [ ] 修正本身是否正確（避免修錯反而引入新問題）
- [ ] 有無新的細節問題（初校未覆蓋處）

**輸出格式：**
```
二校報告（proofreader-second）
初校修正確認：[全部正確 / X 項有誤]
有誤項目：...
新發現問題：...
```

---

### 三校 subagent — `proofreader-final`

**前置條件：** 二校清單已全部處理完畢。

**職責：** 以作者/發布者視角進行最終通讀，模擬讀者/使用者體驗，確認沒有遺漏。

**checklist：**
- [ ] 冷讀（cold read）：從頭讀到尾，語感是否流暢自然
- [ ] 所有前輪標記的問題均已解決
- [ ] 作者、日期、標籤、摘要等 metadata 正確
- [ ] 發布平台的排版格式（Markdown/HTML/富文本）符合預期

**輸出格式：**
```
三校結果（proofreader-final）
狀態：[定稿確認 / 有待處理項目]
待處理（若有）：...
簽核：proofreader-final @ {日期}
```

## Subagent 派發方式

每輪審校均使用 `runSubagent` / Task tool 派發，**不可在同一 agent session 內角色扮演**。

```
範例（初審）：
runSubagent(
  agentName: "content-first-reviewer",
  prompt: """
  你是初審員。以下是待審內容：

  [貼入草稿全文或 diff]

  請依照初審 checklist 完成審查，
  輸出「初審結果」格式的報告。
  """
)
```

每輪 subagent 任務描述中應明確說明：
1. 它的角色與本輪職責
2. 上一輪的審查結果（若有）
3. 當前文章全文
4. 要求的輸出格式

---

## 退修循環

每輪若狀態為「退修」，流程如下：

```
退修 → 原作者修改 → 重新送審（從當輪重審，不需從頭）
       （若修改幅度大，需從初審重新開始）
```

**修改幅度判斷標準：**
- 修改 < 20% 內容 → 從退修的當輪重審
- 修改 ≥ 20% 結構 → 退回初審重新走完整流程

## 快速參考

| 階段 | Subagent 名稱 | 聚焦點 |
|------|--------------|--------|
| 初審 | `content-first-reviewer` | 結構、完整性、基本品質 |
| 複審 | `content-deep-reviewer` | 邏輯、準確性、安全性、深度 |
| 終審 | `content-chief-reviewer` | 全局判斷、發布裁決 |
| 初校 | `proofreader-first` | 細節錯誤逐行掃描 |
| 二校 | `proofreader-second` | 確認修正正確、找新錯誤 |
| 三校 | `proofreader-final` | 作者冷讀、最終簽核 |
