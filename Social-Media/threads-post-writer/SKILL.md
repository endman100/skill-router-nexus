# Threads Post Writer

這個 skill 專門用來為 Threads 社群平台撰寫貼文。

不論話題是科技、產品、趨勢還是知識分享，都能套用統一的撰寫規則，產出風格一致、易讀、利於互動的 Threads 文章。

---

## 第一個段落：此 skill 適用於

- 撰寫科技、AI、產品、趨勢類 Threads 貼文
- 將複雜技術話題轉化為一般大眾可讀懂的文章
- 需要吸引留言互動的知識型貼文
- 希望維持客觀立場、無個人色彩的內容
- 任何需要符合以下「撰寫規則」的 Threads 貼文任務

---

## 撰寫規則（強制執行）

### 結構規則

1. **首段必須有 Hook**：開頭第一段要抓住讀者注意力，直接點出文章核心主題
2. **首段內容**：首段需要讓讀者快速了解文章主題，並且引起興趣。
3. **首段結尾吸引繼續閱讀**：首段的最後一句，必須是引發好奇心、讓讀者想繼續往下看的句子
4. **分段明確**：每個概念獨立一段，段與段之間空一行
5. **遇到「。」必須換行並空一行**：不可將多個句子連接在同一行，也就是每個句子都要雙行分隔，提升可讀性
6. **適時使用條列式**：當資訊有列舉性質時，請用條列方式呈現，提升可讀性
7. **段落數量與長度**：約有 5-9 個段落，視內容而定。每個段落包含多個句子，至少 5 句，建議 5~20 句。
8. **避免數值解釋**：盡量避免使用數字、百分比等具體數值，改用描述性語言解釋實驗或評估結果

### 內容規則

1. **重點放在技術層面**：文章的主要內容應聚焦於技術說明與運作方式
2. **倫理或反思僅出現在最後段落**：如有倫理、社會影響等討論，只能放在最後一段
3. **客觀角度，無個人經驗**：不要以「我」的角度敘述，不帶個人故事或個人觀點
4. **不設角色，不模擬身份**：不要扮演任何人，不要以某個角色說話
5. **讀者非專業領域人士**：避免使用艱深數字、專業術語，以日常語言解釋
6. **不顯示日期或時間**
7. **不放網址**
8. **'.' 後面記得空格，避免成為網址**
9. **不放參考連結**，也不放類似 `[[6]](#__6)` 這類難以閱讀的符號
10. **不使用 LaTeX 數學格式**
11. **不提及中國大陸公司名稱**，但可以提到研究團隊名稱
12. **專有名詞請用縮寫**，如「大語言模型」改為「LLM」，「人工智慧」改為「AI」等
13. **數字請用阿拉伯數字+量化單位**，如「10 萬」改為「100K」，「1 億」改為「100M」等
14. **請用全形標點符號**，如「，」而非「,」，「。」而非「.」，以提升閱讀體驗
15. **句子結尾請省略句號**，以符合社群貼文風格

### 禁用詞彙（完整清單）

以下詞彙完全禁止出現在文章中——

| 禁用詞 |
|--------|
| 大部分人以為 |
| 不是⋯而是⋯ |
| 這根本⋯是在⋯ |
| 愣住了 |
| 更扯的是 |
| 結果呢 |
| 意味著 |
| `--`（雙破折號） |
| `**`（雙星號） |
| 懶人包｜ |
| 是用來被⋯的⋯ |
| 是來被⋯的 |
| 不僅⋯更⋯ |
| 不僅⋯更 |
| 此外 |
| 深入探討 |
| 震撼全球 |
| 想像一下 |
| 對一般使用者來說 |
| `:;、`（這些標點符號） |
| 「」（書名號式引號） |
| 可XX性（可讀性、可擴展性等類似結構） |
| 對⋯來說 |
| 這不是⋯而是⋯ |

## 文章段落架構

```
[首段 Hook]
  → 第一行是完整論文 Title 或主題名稱
  → 第二行是完整論文 Title 或主題名稱的翻譯
  → 點出主題核心
  → 一句話說明結論
  → 最後一句引發好奇心
  → 首段最後加上 👇
  → 首段字數約 100~300 字

[技術說明段落 1]
  → 解釋技術運作方式（白話文）

[技術說明段落 2]
  → 延伸技術重點，可用條列式

[技術說明段落 3（可選）]
  → 更多細節或應用場景

[技術說明段落 x]
  → xxxx

[反思段落（可選）]
  → 僅在此段討論倫理、社會影響等

[快速結論]
```

## 執行流程

1. 接收使用者提供的主題或素材
2. 判斷文章核心訊息與技術重點
3. 依照撰寫規則產生 JSON 結構（見下方「JSON 輸出格式」）
4. 執行 Python 驗證（段落數量 + 段落長度），若不通過則重新產生 JSON
5. 執行 Python 將 JSON 輸出為 preview MD 檔案
6. 輸出 JSON 與 preview MD 路徑

---

## JSON 輸出格式

LLM **必須**輸出以下 JSON 結構——一個 list，紀錄多個段落。每個段落是一個 list，紀錄多個句子：

```json
[
  [
      "第一句。",
      "第二句。",
      "引發好奇心的結尾句。"
  ],
  [
      "第一句",
      "第二句",
      ...
  ],
  ...,
  [
      "第一句。",
      "第二句。"
  ]
]
```

**限制**：
- 必須有 5–9 個段落
- 每個段落的至少 5 句，建議 5~20 句

---

## Python 驗證流程

在 LLM 產生 JSON 後，**必須**執行驗證腳本：

```
python {SKILL_PATH}/validate.py <json_file_path>
```

- 驗證通過：印出 `驗證成功：共 N 段`，exit code 0
- 驗證失敗：印出所有 `[錯誤]` 訊息，exit code 1

若驗證失敗，必須將所有錯誤訊息回饋給 LLM，要求修正後重新產生 JSON，再次驗證，直到通過為止。

驗證規則（定義於 `{SKILL_PATH}/validate.py`）：
1. 頂層必須是 list
2. 段落數量：5–9 段
3. 每個段落至少 5 句

---

## Preview MD 輸出規則

驗證通過後，**必須**執行 preview MD 產生腳本：

```
python {SKILL_PATH}/to_preview_md.py <json_file_path> [base_filename]
```

- `json_file_path`：已通過驗證的 JSON 檔案路徑
- `base_filename`：輸出檔案的基底名稱（不含副檔名，可選；若省略則以 JSON 檔名為基底）

輸出檔案會放在與 JSON 同目錄，檔名格式為 `<base_filename>-preview.md`。

> ⚠️ **Preview MD 檔案規則（強制）**
>
> - 檔名**必須**含 `-preview` 後綴（如 `my-post-preview.md`）
> - 此檔案**不可手動修改**，只能作為排版預覽用途
> - 若要更新內容，必須修改來源 JSON → 重新執行 `validate.py` → 重新執行 `to_preview_md.py`
> - 可在 CI 或工作流中設定 `.gitattributes` 或 pre-commit hook 防止直接 commit 對此檔案的手動修改

---

## 注意事項

- 若使用者提供的主題資訊不足，先詢問補充，再開始撰寫
- 若主題橫跨多個技術面向，優先選擇最核心的一個聚焦，其餘以條列方式帶過
- 不要在文章內解釋這些規則，直接按規則輸出文章即可
